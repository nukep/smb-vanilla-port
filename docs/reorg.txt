Drawbacks of current interim design
- various ".c" files are #include'd, such as common.c, area.c, etc. This was a workaround until a better design is figured out.
- a SMB_STATE singleton is used everywhere. this is necessary for a library approach, but doesn't work well for embedded ports where we actually want everything to be global variables.
- the use of callbacks is mandatory. this isn't too great if someone wants to bypass the indirection of callbacks.



Desired API

- have an interface layer that can have its implementation swapped out
  - allows for (more customizable) embedded targets
  - the callback and SMB_state system becomes one interface layer implementation
  - macros such as RAM are moved to this new interface layer
- ROM arrays and constants should be relocatable (to support SMB1 FDS, PAL)
  - possible strategies for ROM arrays:
    - load them directly from in-memory ROM
    - use pre-extracted data
