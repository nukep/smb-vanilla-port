  ---
  Reorganization Plan

  Problem Summary

  All three drawbacks in reorg.txt stem from the same root: the game engine code
  (common.c, area.c, etc.) is tightly coupled to SMB_STATE and callbacks
  through macros defined in smbcore.h. The #included .c files are a symptom of
  this — they can't be compiled separately because they depend on context
  (SMB1_MODE, RAM, draw_tile, etc.) injected by the including translation unit.

  ---
  Proposed Structure

  Step 1: Extract an interface layer (smbcore/interface.h)

  Split smbcore.h into two concerns:

  - smbcore.h — keep utility types, math macros (LOAD_16, ADD_16_16, etc.),
  ppureg/ppu_state/sprite structs, and non-interface helpers. No references to
  SMB_STATE.
  - smbcore/interface.h — new file, selected at build time. Defines the macros
  the game code actually uses to touch hardware: RAM, PPURAM, CHRROM, AreaData,
  EnemyData, draw_tile, apu_write_register, joy1, joy2, etc.

  Two reference implementations:

  ┌───────────────────────────────┬──────────────────────────────────────────┐
  │             File              │               Description                │
  ├───────────────────────────────┼──────────────────────────────────────────┤
  │                               │ Current behavior: all macros go through  │
  │ smbcore/interface_callbacks.h │ SMB_STATE->... and                       │
  │                               │ SMB_STATE->callbacks.*                   │
  ├───────────────────────────────┼──────────────────────────────────────────┤
  │                               │ Embedded-friendly: macros resolve to     │
  │ smbcore/interface_globals.h   │ global arrays/variables; callbacks       │
  │                               │ replaced with extern function            │
  │                               │ declarations                             │
  └───────────────────────────────┴──────────────────────────────────────────┘

  smb1.cpp and smb2j.cpp #include the chosen interface before including the .c
  files. The game code itself doesn't change.

  This means the singleton SMB_STATE and thread_local pattern become concerns of
  interface_callbacks.h only — they disappear from the embedded path entirely.

  Step 2: Rename #included .c files to .inc

  Rename common.c, area.c, common_sound.c, smb1only.c, smb2jonly.c to .inc (or
  .c.inc). No code changes required; just eliminates confusion about whether
  they're standalone compilation units. The build system already ignores them.

  Step 3: ROM array relocatability

  smb_romarrays.h and smb2j_romarrays.h currently hardcode data arrays. For
  FDS/PAL support, the ROM addresses of these arrays shift.

  Two categories of data in those files:

  - Code-like data (enemy speeds, physics constants, etc.) — keep hardcoded. Per
  decisions.md this is intentional.
  - Level-editable data (area pointers, music pointers, tile lookups used by the
  renderer) — convert to ROM-offset indirection, similar to how
  AreaData/EnemyData/MusicData are already handled. Store the ROM base address
  at load time; arrays become rom_ptr(base + offset) lookups. This naturally
  accommodates FDS and PAL with different base addresses.

  The ROM loading code in smbcore.cpp (load_smb1, load_smb2j) already has a
  natural place to set these base addresses.

  Final target

  Two libraries, libsmb1 and libsmb2j, each independently self-contained and
  exposing the same mario.h public API. A game-agnostic interface layer controls
  how the game code reaches hardware abstractions, making embedded and
  non-callback builds possible. The libraries cannot be linked together.

  ├──────────┼───────────────────────────────────────────────────────────────┤
  │ libsmb2j │ smb2j.cpp, smbcommon.cpp, common.c, area.c, common_sound.c,   │

  ---
  Phase 1 — Context headers, interface layer, standalone TUs

  1a. Split smbcore.h into two parts.

  smbcore.h currently mixes two concerns: utility code (math macros, ppureg,
  ppu_state, byte/ushort typedefs, RamByteArray, helper functions like
  CONCAT11/CARRY1/etc.) and the interface layer (RAM, PPURAM, draw_tile, joy1,
  apu_write_register, SMB_STATE, etc.). Split them:

  - smbcore.h — keep only the game-agnostic utilities: types, math macros,
  ppureg/ppu_state/sprite, assert_*, RamByteArray, jmpengine_overflow, CONCAT11,
  SWAP, ADD_16_16, etc. No reference to SMB_STATE or callbacks.
  - smbcore/interface.h — thin selector that #includes one of the two
  implementations below.

  1b. Create two interface implementations.

  smbcore/interface_callbacks.h — extracts what's currently in smbcore.h that
  goes through SMB_STATE: RAM, PPURAM, CHRROM, AreaData, EnemyData, MusicData,
  draw_tile, can_draw_tile, apu_write_register, apu_end_frame, joy1, joy2,
  announce_main_scroll, transfer_sprite_data, smb2j_load_games_beaten,
  smb2j_save_games_beaten, update_pattern_tables, PPU_STATE, rom_ptr,
  joystick_strobe, all APU_REG inlines, SMB_STATE extern, etc.

  smbcore/interface_globals.h — new file. Same macro/function names, but
  implemented as bare globals and extern function declarations. No SMB_STATE
  struct. No callbacks. RAM(x) becomes smbglobals_ram[x]. Intended for embedded
  targets where everything is global.

  1c. Create per-game context headers.

  #include "smbcommon.h"
