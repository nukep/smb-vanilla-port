● I've rewritten the plan around your answers. The key reframing:

  The big inversion. My earlier "rename the shared files to .cpp" was exactly
  wrong for your goal — it would have silently destroyed the C-compilability
  invariant. The plan now makes C the default build: the core files keep .c, so
  meson's extension-based dispatch proves "compiles as C" continuously, for
  free. C++ is demoted to an opt-in luxury (force-compile the same .c files as
  C++ under CHECK_ARRAY_BOUNDS for the RamByteArray bounds-checker).

  The design now centers on one idea, not a symbol list. smbcore.h blends three
  concerns — pure utility (A), the interface contract (B), and the callback-impl
  internals incl. C++-isms (C). The whole reorg is separating A/B/C so the core
  depends only on A + the declared interface, with C confined to one swappable,
  recompile-time-selected implementation. The phases serve that.

  Your "either interface must be C" answer raised the bar usefully. It turns the
  callback interface's own C++-isms into debt this reorg must clear, not just
  the bulk game code. I isolated exactly three: thread_local SMB_STATE, ppu.h's
  ushort &v reference, and the RamByteArray/{0}-aggregate pair. Settling those
  three is now framed as the spine of Phase 1, with file-splitting as
  bookkeeping around it.

  Symbol enumeration demoted. The cross-TU symbol table moved to an appendix
  labeled "mechanical, not design," per your request to focus on the design.

  Two things I deliberately left as flagged open decisions rather than guessing,
  because they're yours to make:
  - Executables/testrunner after the split — two binaries (recommended) vs. one

  - RamByteArray-vs-RAM include ordering — a real layering pitfall since the
